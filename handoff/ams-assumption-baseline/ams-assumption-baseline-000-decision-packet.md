## 0. 이 패킷이 정하지 않는 것 (경계 명시)

- `[AMS 분석 가정 — 검증 필요]` 항목의 **실제 답은 전부 사람이 정한다.** 이 패킷은 각 항목의 **분류·의존·역전 비용·격리 가능성·권고 시점만** 담는다.
- 통합본 13장 §J는 **미결 Open 영역**이다(CLAUDE.md §2 / 강제 룰 5). 따라서 이 패킷은 가정의 답을 추측으로 메우지 않으며, **프로세스(태그 전환·게이트·문구 처리)에 대한 사람 승인 4건(D1~D4)만** blocker로 올린다. 가정의 실제 답(14항목 각각의 확정값)은 **이 패킷 승인 이후 항목별 입력 단계**의 몫이다.

## 1. 배경 — 왜 baseline 전환인가 (사람이 이미 정한 방향)

- 통합본 "다음 단계 1. 5단계 AMS 피드백"(84행)·작성 원칙 §0.3(128행)·§01 결정 요약(111행)은 **AMS 실무자 외부 검증 채널**을 전제했다. 현실에는 그 채널이 지금 없다.
- 사용자 결정(2026-06-13): **사용자가 항목별로 잠정 확정하고, 추후 팀 공유 시 재검증·개선**하는 방식으로 전환.
- 안전 조건(사용자 합의): ① 태그를 지우지 않고 "사용자 잠정 확정(날짜) — 팀 리뷰 시 재검증" 형태로 전환 ② 항목별 결정 기록(`handoff/decisions/`) ③ 팀 리뷰 게이트 명시(마이그레이션/ETL 설계 착수 전, 늦어도 본개발 진입 전) ④ 역전 비용 큰 항목은 어댑터/인터페이스 뒤로 격리.
- 진행 방식은 **pull**: 현재 액티브 큐(result-topic 분리(T4-2)·메시지 키 정의(T4-4)·계약 문서 갱신(T4-5))는 §J에 안 막힘. 첫 차단 지점은 Rule Engine(T2-1) 착수 전, Job 결재/Validation 설계 전, 도메인 DDL(D-2 후) 전, 마이그레이션/ETL 설계 전.

## 2. 현황 — §J 가정이 통합본 baseline에서 차지하는 위상

- 통합본은 §J 가정을 "1차 결정은 모두 변경 가능 전제"(작성 규칙 99~100행) 위에서 **잠정 baseline으로 박아둔 상태**다. 즉 대부분의 가정은 이미 통합본 본문에 "1차 결정 방향"으로 결정값이 들어가 있고, §J 태그는 "그 결정이 AMS 검증을 못 받았다"는 **꼬리표**다.
- 따라서 baseline 전환의 실질은 **새 결정을 만드는 게 아니라**, 기존 잠정 baseline에 "누가·언제 잠정 확정했고 무엇을 재검증해야 하는지"를 **기록 가능한 형태로 승격**하는 것이다.
- ROADMAP은 이 게이팅을 **AMS 5단계 검증 게이트(D-1, BLOCKED)**로 추적한다(§17 D-1: "§J ↔ ADR/컴포넌트 대조해 실제 게이팅 대상 목록 작성"). 이 패킷의 §3 차단 Track 분석이 D-1의 next_action을 사실상 수행한다.

## 3. §J 14항목 전수 분석

> §J 원문(3277~3290행)은 14개 불릿. 아래 번호는 원문 순서.
> 각 항목: (a) 가정 요약 (b) 의존하는 통합본 baseline 결정 (c) 차단 Track (d) 역전 비용 (e) 격리 가능성 (f) 권고 결정 시점.
> 역전 비용 판정 기준: **높음**=도메인 모델·ETL·룰 엔진에 깊이 박혀 잘못 가정 시 재작업 큼 / **중간** / **낮음**=어댑터 경계 뒤라 싸게 뒤집힘.

### J-01. 원격 쉘 실행 권한 모델 (§3.3)

- (a) Agent 원격 명령/스크립트 실행의 권한·통제 모델. 통합본 baseline = **화이트리스트 + 사전 등록 스크립트 + 시간 상한 + 출력 크기 제한 + 감사 로그**(§4.1.7, 482행).
- (b) 의존 baseline: §4.1.7 원격 명령 실행 통제 모델 / §13 §F "원격 명령 화이트리스트 운영자 자율성"(3251행, 운영 적합성 Open).
- (c) **차단 Track**: 현재 액티브 큐 무차단. SCRIPT_JOB 실행은 Phase 0에서 사전 등록 스크립트 모델로 이미 동작(데모 spec). 신규 임의 원격 쉘(ad-hoc) 기능은 Phase 1 Track에 작업 항목 없음 → **마이그레이션 설계 전까지 무차단**.
- (d) **역전 비용: 낮음** — 화이트리스트/시간상한/출력제한은 실행 게이트(정책 레이어)라 정책 객체 교체로 흡수.
- (e) 격리: 해당(정책 게이트로 이미 격리). 추가 어댑터 불요.
- (f) **권고 시점: 유보 가능** — 운영자 자율성 폭(§F Open)은 운영 정보가 와야 정밀해짐. 잠정 baseline 유지로 충분.

### J-02. 서브시스템 모델의 OpenITSM/MSP+ 의존도 (§4.1.1)

- (a) AMS 모니터링 대상 "서브시스템" 모델이 OpenITSM(구, 폐기 대상)·MSP+(신 ITSM)에 얼마나 묶여 있는가. 통합본 baseline = **AMS 서브시스템 + SMS 장비 속성 통합**(§4.1.1, 289행), OpenITSM **폐기**·MSP+ **어댑터 옵션**(§4.1.10, 560~561행).
- (b) 의존 baseline: 모니터링 대상 도메인 모델(§4.1.1·§4.2.1 "모니터링 대상/CI" 통합 이슈) / 마이그레이션 "모니터링 대상=Agent 등록(KMC 기반) `[AMS 분석 가정]`, OpenITSM/MSP+ 의존도 검증 후 결정"(§11.3, 2952행).
- (c) **차단 Track**: 액티브 큐 무차단. **도메인 DDL(모니터링 대상 entity 모델) 설계 전 + 마이그레이션 ETL 설계 전 차단** — 서브시스템이 ITSM에 강결합이면 entity 스키마·ETL 매핑이 달라짐.
- (d) **역전 비용: 높음** — 모니터링 대상 도메인 모델·ETL에 깊이 박힘. 서브시스템 정의가 틀리면 entity DDL과 이관 매핑 재작업.
- (e) 격리: **부분 가능** — 외부 ITSM 연계는 Asset Adapter(§7.6, 2176행)로 격리되나, **서브시스템 자체의 entity 모델은 도메인 코어라 어댑터 뒤로 밀 수 없음**. 어댑터로 흡수되는 건 "ITSM 연계 여부"뿐, "서브시스템 데이터 모델"은 아님.
- (f) **권고 시점: 도메인 DDL 전** — 모니터링 대상 entity DDL 착수 직전 잠정 확정 필요. 그 전까지는 유보 가능.

### J-03. Log Job 동작 모델 / Log raw 저장 / Windows Event Log (§4.1.3)

- (a) AMS Log Job의 동작 방식, raw 로그 저장 여부, Windows Event Log 지원 범위. 통합본 baseline = **Script Agent의 LOG Job 유형 통합 + 옵션 A(BE-side 매칭) + OpenSearch raw 보존 baseline**(§4.1.3, 344~346행). 원문 표에 AMS의 raw 저장·Windows Event Log는 `?`(미확정) 다수.
- (b) 의존 baseline: 로그 도메인(§4.1.3) / 데이터 모델 "로그=OpenSearch 저장"(§4.2.1) / 옵션 A 확정(§6.x). **단 LOG_JOB 수집 흐름은 Phase 0에서 이미 구현·검증됨**(`docs/features/log-job-collection.md`, e2e 60/0/0).
- (c) **차단 Track**: 액티브 큐 무차단. LOG_JOB occurred_at(T3-7)·log pipeline은 통합본/envelope spec 기반이라 §J 가정과 독립. AMS 특유의 raw 저장 정책·Windows Event Log 매핑은 **마이그레이션 ETL 설계 전까지 무차단**(신규 Phase 1 작업 항목에 AMS Log 이관 없음).
- (d) **역전 비용: 중간** — 로그 처리 흐름(옵션 A)은 확정·구현됨. raw 저장 보존 기간·Windows Event Log 소스 매핑은 ILM 정책·수집 어댑터 레벨이라 도메인 코어보다 얕음.
- (e) 격리: **가능** — raw 보존은 OpenSearch ILM 정책, Windows Event Log는 Agent 수집 소스 어댑터로 격리.
- (f) **권고 시점: 마이그레이션 설계 전** — Phase 1 구현은 §J와 독립으로 진행 가능. 잠정 확정은 AMS 로그 이관 ETL 직전이면 충분.

### J-04. Drools 7.x 룰 복잡도 (§4.1.4, §11.3)

- (a) AMS 기존 Drools 7.x 룰의 복잡도(= 8.x 재작성 비용). 통합본 baseline = **Drools 8.x 채택, 7.x→8.x 재작성**(§4.1.4, 379행) + "DRL 룰 8.x 호환 재작성 + 자동 변환 도구 + 수동 검증"(§11.3, 2963행). §11.8은 이를 **마이그레이션 단일 최대 비용**으로 명시(3118행).
- (b) 의존 baseline: Rule Engine(Drools 8.x) 채택 결정 / 마이그레이션 비용 산정(§11.3·§11.8).
- (c) **차단 Track**: **Rule Engine(T2-1) 착수 전 부분 차단 + 마이그레이션 ETL 설계 전 차단.** T2-1은 신규 Drools 8.x 엔진을 **새로 빌드**하는 작업이라 7.x 룰 복잡도 없이도 엔진 골격은 진행 가능. 단 룰 재작성/이관 비용 산정과 룰 데이터 모델 검증은 복잡도 입력 필요.
- (d) **역전 비용: 높음** — 룰 엔진·ETL에 직결. 7.x 룰이 예상보다 복잡하면 자동 변환 도구 범위·수동 검증 공수·룰 데이터 모델이 모두 흔들림.
- (e) 격리: **제한적** — Drools 8.x 엔진 자체는 Rule Engine 컴포넌트로 격리되나, **7.x→8.x 룰 변환 비용·룰 표현력 요구는 격리 불가**(룰의 본질적 복잡도라 어댑터 뒤로 못 숨김).
- (f) **권고 시점: Rule Engine(T2-1) 착수 전** — 엔진 골격은 가정 없이 가능하나, 룰 데이터 모델·복합 이벤트 표현력 요구는 T2-1 설계 입력. 늦어도 마이그레이션 설계 전 확정.

### J-05. 복합 이벤트 메커니즘 (§4.1.4, §11.3)

- (a) AMS 복합 이벤트(여러 이벤트 조합 → 상위 이벤트) 정책의 실제 메커니즘. 통합본 baseline = **Rule Engine 복합 이벤트**(§4.1.4 "복합 이벤트 정책" ●) + "복합 이벤트 메커니즘 검증 후 ETL `[AMS 분석 가정]`"(§11.3, 2949행).
- (b) 의존 baseline: Rule Engine 복합 이벤트 처리 설계 / 이벤트 도메인 모델.
- (c) **차단 Track**: **Rule Engine(T2-1) 착수 전 차단(핵심).** 배경에서 사용자가 지목한 "Rule Engine 전 첫 차단 지점(복합 이벤트)"이 바로 이 항목. 복합 이벤트 메커니즘이 stateless 룰로 표현되는지/상태 추적이 필요한지가 엔진 설계(stateless Drools 8.x baseline)를 좌우.
- (d) **역전 비용: 높음** — 룰 엔진 아키텍처 자체에 박힘. 복합 이벤트가 상태 기반이면 stateless baseline(§4.5 "Drools 8.x stateless")을 재검토해야 함.
- (e) 격리: **불가** — 복합 이벤트 처리 모델은 Rule Engine 코어 설계라 어댑터 뒤로 격리 불가.
- (f) **권고 시점: Rule Engine(T2-1) 착수 전(필수)** — T2-1 설계의 직접 입력. 잠정 확정 묶음의 1순위.

### J-06. Job 결재 활용도 / Validation 부하 (§4.1.5)

- (a) AMS Job 결재 프로세스가 실제로 얼마나 쓰이는가 + Validation Check의 운영 부하. 통합본 baseline = **결재 프로세스 옵션화 확정**(§4.1.5, 417행) + **Validation 권한 분리**(418행). §13 §F "결재 프로세스 사내 활용도"(3229행)·"Validation 권한 분리 운영상 영향"(3252행) Open.
- (b) 의존 baseline: 결재 어댑터(T3-4) / Validation Service(T3-3) 설계 범위·우선순위.
- (c) **차단 Track**: **Job 결재/Validation 설계(T3-3·T3-4) 전 차단.** 두 Track 모두 현재 액티브 큐 밖(Track 3). 활용도가 낮으면 옵션화로 충분, 높으면 설계 우선순위·범위 상향.
- (d) **역전 비용: 중간** — 결재는 이미 어댑터(Approval Adapter, §7.6)로 격리됨 → 활용도 가정이 틀려도 어댑터 활성/비활성 + 우선순위 조정으로 흡수. Validation은 권한 분리 모델이 도메인에 일부 걸려 중간.
- (e) 격리: **가능** — 결재는 Approval Adapter + ConfigMap enabled 플래그로 완전 격리. Validation은 Service 경계로 부분 격리.
- (f) **권고 시점: Job 결재/Validation 설계(T3-3·T3-4) 전** — 그 전까지 유보 가능(어댑터 격리로 안전).

### J-07. AMS 결재 모델이 외부 시스템 통합인지 자체 결재인지 (§6.2, §11.3)

- (a) AMS 결재가 Knox Approval/BCMS 등 외부 결재 시스템 통합인지, AMS 자체 결재 테이블인지. 통합본 baseline = **Approval Adapter + 외부 결재 시스템**, "자체 결재인지 외부인지 검증 필요 `[AMS 분석 가정]`"(§11.3, 2947행). §7.6 결재 어댑터 = "사이트별(Knox Approval/BCMS/자체)"(2175행).
- (b) 의존 baseline: 결재 어댑터(T3-4) 구현 형태 / 결재 데이터 ETL(자체 결재면 결재 테이블 이관 필요, 외부 통합이면 연계만).
- (c) **차단 Track**: **결재 어댑터(T3-4) 설계 전 + 마이그레이션 ETL 설계 전 차단.** T3-4 blocked_by="외부 결재 시스템 정보"(ROADMAP §12)와 직결.
- (d) **역전 비용: 중간** — 어댑터 경계는 잡혀 있으나(낮춤 요인), 자체 결재면 결재 테이블 DDL·ETL이 추가로 필요(높임 요인) → 중간.
- (e) 격리: **가능(연계 측면)** — 외부 통합/자체 결재 분기를 Approval Adapter 구현체 교체로 흡수. 단 자체 결재일 때의 결재 도메인 테이블은 어댑터 밖 코어.
- (f) **권고 시점: 결재 어댑터(T3-4) 설계 전** — J-06과 함께 묶어 잠정 확정 권고.

### J-08. Knox Portal Mail 연계 (§4.1.6)

- (a) 통보 메일을 Knox Portal Mail로 보내는 연계 방식. 통합본 baseline = **Notification Email 어댑터(adapter: knox-mail)**(§4.1.6 통보 어댑터 + §7.6, 2171·2189행).
- (b) 의존 baseline: Notification Service 채널 어댑터 4종(T3-1).
- (c) **차단 Track**: 액티브 큐 무차단. Notification Service(T3-1)는 채널 어댑터 계약 테스트로 진행 → **마이그레이션/사이트별 연계 검증 전까지 실질 무차단**.
- (d) **역전 비용: 낮음** — 통보 채널 어댑터(§7.6) + ConfigMap enabled 플래그로 완전 격리. Mail 시스템 차이는 어댑터 교체로 흡수.
- (e) 격리: **완전 가능** — Notification Email 어댑터.
- (f) **권고 시점: 유보 가능** — 어댑터 격리로 가장 싸게 뒤집힘.

### J-09. Agent 부서 책임 분산 / 자동 복구 (§4.1.7)

- (a) AMS의 "부서 책임 분산 → Unknown Agent 누적" 운영 문제 해결 방향 + 신규 Agent 자동 복구. 통합본 baseline = **중앙 운영자 일괄 제어(부서 책임 분산 해결)**(§4.1.7, 481행) + **자동 복구 신규 도입**(480행). §13 §F "AMS 부서 책임 분산 해결 방향"(3250행)·"Agent 자동 복구 실용성"(3249행) Open.
- (b) 의존 baseline: Agent State Service(T2-5) / 권한 모델(중앙 일괄 제어).
- (c) **차단 Track**: 액티브 큐 무차단. Agent State Service(T2-5)는 heartbeat infra 기반이라 §J 독립. 부서 책임 분산은 권한 모델(인증/인가 T1-2) 설계와 연관되나 baseline(중앙 일괄)으로 진행 가능 → **마이그레이션/운영 정책 확정 전까지 무차단**.
- (d) **역전 비용: 낮음** — 중앙 일괄 제어 권한 모델 + 자동 복구는 운영 정책/State Service 레벨. 부서 책임 분산 해결은 권한 모델 설정값.
- (e) 격리: 가능(권한 정책 + State Service 경계).
- (f) **권고 시점: 유보 가능** — 운영 적합성(§F Open)은 운영 정보 후 정밀화.

### J-10. Manager 프로파일 변경 차단 (§4.1.8)

- (a) AMS의 "Manager가 임의 부서 프로파일 변경 가능"이라는 운영 문제 차단. 통합본 baseline = **운영 문제 3가지 명시적 해결**(§4.1.8, 513행) + "임의 부서 프로파일 변경 차단"(§4.1.5, 408행).
- (b) 의존 baseline: 권한 체계(AMS 4단계: Admin/Manager/QA/User) / 인증·인가(T1-2).
- (c) **차단 Track**: 액티브 큐 무차단. 인증/인가(T1-2)는 Phase 1 전체 TODO지만 RBAC 권한 매트릭스 설계 시 반영. §J 가정은 "차단해야 한다"는 방향이 이미 확정 → **인증/인가 권한 매트릭스 설계 시 흡수, 별도 차단 없음**.
- (d) **역전 비용: 낮음** — RBAC 권한 규칙(데이터 접근 권한). 권한 정책 객체로 표현.
- (e) 격리: 가능(RBAC 정책 레이어).
- (f) **권고 시점: 유보 가능** — baseline(차단)이 명확하고 RBAC로 흡수.

### J-11. ACS 역할 (§4.1.10)

- (a) AMS 단독 외부 시스템 "ACS"의 역할·연계 필요성. 통합본 baseline = **ACS: AMS 단독, 검증 필요**(§4.1.10, 571행) — baseline 미확정에 가까움(역할 불명).
- (b) 의존 baseline: 외부 시스템 연동 매트릭스(§4.3) / Asset/ITSM 어댑터 우선순위.
- (c) **차단 Track**: 액티브 큐 무차단. ACS는 어떤 Phase 1 Track의 blocked_by에도 없음 → **마이그레이션/사이트별 연계 설계 전까지 무차단**.
- (d) **역전 비용: 낮음** — 외부 연계 어댑터(§7.6 패턴) 추가/생략으로 흡수. 역할 판명 후 어댑터 1개 추가 여부 결정.
- (e) 격리: 완전 가능(외부 시스템 어댑터).
- (f) **권고 시점: 유보 가능** — 역할 정보 입수까지 baseline 미정 유지. 사이트별 연계 우선순위(§F, 3141행)와 함께.

### J-12. AMS 이벤트 테이블 스키마 (§4.2)

- (a) AMS 이벤트 테이블의 실제 스키마(통합본은 "테이블명 미확정"으로 표기). 통합본 baseline = **이벤트/Alert/Incident: AMS+SMS 통합 우선 + Incident 신규**(§4.2.1, 581행), AMS 이벤트 테이블명 미확정.
- (b) 의존 baseline: 이벤트 도메인 DDL / Alert·Incident 도메인 / 이벤트 이력 ETL(§11.3 "이벤트 등급/라이프사이클" 이력 재구성).
- (c) **차단 Track**: **도메인 DDL(이벤트 도메인) 설계 전 + 마이그레이션 ETL 설계 전 차단.** Alert Processor(T2-2)·Incident Service(T2-3)는 신규 도메인 모델을 새로 정의하므로 AMS 스키마 없이 신규 설계 진행 가능, 단 이관 매핑은 차단.
- (d) **역전 비용: 높음** — 이벤트 도메인 DDL·이력 ETL에 직결. AMS 이벤트 스키마를 잘못 가정하면 이관 매핑·이력 재구성 재작업.
- (e) 격리: **제한적** — 신규 이벤트 도메인 모델은 AMS 스키마에 종속되지 않게 설계 가능(통합 우선), 하지만 **이관 ETL 매핑은 실제 AMS 스키마 의존**이라 격리 불가.
- (f) **권고 시점: 도메인 DDL 전(이력 ETL은 마이그레이션 설계 전)** — 신규 도메인 설계는 가정 없이 가능하나, ETL 매핑 직전 잠정 확정 필요.

### J-13. AMS 사용자 테이블 단일화 모델 (§4.2)

- (a) AMS의 `TB_USER`+`tb_user` 이중화를 단일 사용자 테이블로 통합하는 모델. 통합본 baseline = **단일 사용자 테이블 + 외부 동기화 캐시 분리**(§4.2.1, 586행 / §4.2.4, 617행) + "AMS 단일 테이블 모델 차용 — ETL 단순"(§11.3, 2953행).
- (b) 의존 baseline: 사용자/권한 도메인 DDL / 인증·인가(T1-2) / 사용자 ETL.
- (c) **차단 Track**: 액티브 큐 무차단. 인증/인가(T1-2)·사용자 도메인 DDL 설계 전 부분 연관. baseline("단일 테이블+캐시 분리")이 명확하고 §11.3이 "ETL 단순"으로 판정 → **도메인 DDL 전 경미 차단, ETL은 마이그레이션 설계 전**.
- (d) **역전 비용: 중간** — 사용자 도메인 DDL에 박히나, AMS 이중화→단일화 방향이 명확하고 ETL이 단순으로 평가됨. 외부 동기화 캐시 분리 경계가 잡혀 있어 중간.
- (e) 격리: **부분 가능** — 외부 동기화(Knox)는 캐시/어댑터로 분리. 단일 사용자 테이블 자체는 도메인 코어.
- (f) **권고 시점: 도메인 DDL 전** — baseline이 명확해 잠정 확정 부담 낮음.

### J-14. AMS 표준 모니터링 차별화 기능 (§4.1.10)

- (a) AMS의 "표준 모니터링 항목 매칭" 차별화 기능의 실체. 통합본 baseline = **Rule Engine + 카탈로그로 흡수**(§11.3 "표준 모니터링 항목"·F10, 2951·2959행) + "표준 모니터링 항목 매칭"(§4.1.5, 405행). §13 §H 등에서 차별화 기능 = 마이그레이션 비용 직접 영향(§11.8, 3116행).
- (b) 의존 baseline: Rule Engine(T2-1) + 사내 표준 카탈로그 / ETL 비용 산정.
- (c) **차단 Track**: **Rule Engine(T2-1) 착수 전 부분 차단 + 마이그레이션 ETL 설계 전 차단.** 차별화 기능이 룰 엔진으로 표현 가능한지가 T2-1 범위·카탈로그 모델에 영향.
- (d) **역전 비용: 높음** — 룰 엔진·카탈로그·ETL에 박힘. §11.8이 "표준 모니터링 차별화 기능 검증되어야 ETL 비용 정확"으로 명시.
- (e) 격리: **제한적** — 카탈로그는 ConfigMap/파일 import로 일부 격리 가능하나, 차별화 기능의 룰 표현 요구는 Rule Engine 코어.
- (f) **권고 시점: Rule Engine(T2-1) 착수 전** — J-04·J-05와 같은 룰 엔진 묶음.

### 5.3 팀 리뷰 게이트 기록 위치

- 제안: 통합본 **다음 단계 1번(84행) 재서술** — "5단계 AMS 피드백"을 "AMS 분석 가정 — 사용자 잠정 확정 + 팀 리뷰 재검증 게이트"로 바꾸고, 게이트 시점을 명시: **마이그레이션/ETL 설계 착수 전 + 늦어도 본개발 진입 전**(§11.8 3116행과 정합).
- ROADMAP 측: §17 D-1(AMS 5단계 검증 게이트)의 next_action을 "§J↔ADR 대조"에서 "**항목별 잠정 확정 진행 + 팀 리뷰 게이트 추적**"으로 갱신 제안(별도 ROADMAP 갱신 작업).
- 근거: 안전 조건 ③. 게이트 시점은 역전 비용 높은 5개 항목의 차단 Track(T2-1 / 도메인 DDL / 마이그레이션)과 정렬.

### 5.4 "5단계 AMS 피드백" 문구 처리 (84·111·128행)

외부 피드백 전제 문구가 3곳. 두 후보:

- **후보 A — 재서술(권고)**: 세 문구를 "사용자 잠정 확정 + 팀 리뷰 재검증" 방식으로 직접 고쳐 씀.
  - 84행: "5단계 AMS 피드백" → "AMS 분석 가정 잠정 확정 + 팀 리뷰 재검증(마이그레이션/ETL 설계 전·본개발 진입 전 게이트)".
  - 111행/128행: "5단계 피드백 시 검증"/"5단계 피드백 후 갱신 대상" → "사용자 잠정 확정 후 팀 리뷰 재검증 대상".
  - 장점: 통합본 본문이 현실 프로세스와 일치. 단점: 본문 수정이라 통합본 update-proposal + 사람 승인 + codex-gate 필요.
- **후보 B — 주석 보존**: 원 문구 유지 + 각주/박스로 "2026-06-13 결정: 외부 채널 부재로 사용자 잠정 확정 방식 전환, 상세 `handoff/decisions/ams-assumption-decisions.md`" 추가.
  - 장점: 이력 보존, 외부 인용 앵커 안전. 단점: 본문에 stale 문구 잔존.

> **통합본 본문 수정은 analyzer가 직접 하지 않는다**(강제 룰 4). 후보 확정 시 `handoff/ams-assumption-baseline/통합본-update-proposal-ams-assumption-baseline.md` 별도 제안서로 올린다. (이 패킷은 그 전 단계 — 후보 제시만.)

## 7. 결정 필요 사안 (blockers — 사람 입력 대기, analyzer 자율 결정 금지)

> 아래는 **프로세스 승인 4건(D1~D4)** 만. 가정의 실제 답(14항목 확정값)은 이 패킷 승인 후 항목별 입력 단계의 몫이며 여기 D-목록에 넣지 않는다.

- **D1 — 진행 방식 확정**: 일괄 14개 지금 확정 vs pull 묶음별 진행.
  - **권고: pull 묶음별.** 근거: §4.1대로 지금 액티브 큐(T4-2/T4-4/T4-5)는 §J에 무차단 — 14개 일괄 확정의 강제 사유 0. 첫 실질 게이트는 Rule Engine(T2-1) 전 3개(J-04·J-05·J-14). 일괄 확정은 토큰·검증 비용만 키우고 재검증 대상만 늘림.
- **D2 — 태그 전환 형식 승인**: §5.1 `[AMS 분석 가정 — 사용자 잠정 확정 YYYY-MM-DD, 팀 리뷰 재검증]` 형식 채택 여부(작성 규칙 §0.3 변경).
  - **권고: 채택.** 근거: 안전 조건 ①(태그 보존)·②(기록)와 정합. 형식 변경은 통합본 본문 수정이므로 update-proposal 경유.
- **D3 — 팀 리뷰 게이트 시점 승인**: "마이그레이션/ETL 설계 착수 전 + 늦어도 본개발 진입 전" 게이트 채택 여부 + 통합본 다음 단계 1번에 기록 여부.
  - **권고: 채택.** 근거: §11.8(3116행)이 이미 "AMS 검증되어야 ETL 비용 정확"으로 동일 시점 명시 — 정합. 역전 비용 높은 5개 차단 Track과 정렬.
- **D4 — "5단계 AMS 피드백" 문구 처리**: 후보 A(재서술) vs 후보 B(주석 보존).
  - **권고: 후보 A(재서술).** 근거: 외부 채널 전제가 사라졌으므로 본문이 현실과 일치하는 게 drift 방지에 유리. 단 통합본 본문 수정이라 별도 update-proposal + codex-gate 필요(이 패킷에선 후보 제시까지).

> 추가 멈춤 사유: 통합본 13장 §J는 **미결 Open 영역**(강제 룰 5). 이 패킷은 §J 가정의 답을 추측하지 않았으며, 가정의 실제 확정은 D1 진행 방식 승인 후 사람 입력으로만 진행한다.
